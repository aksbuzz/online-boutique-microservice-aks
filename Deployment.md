# Online Boutique Deployment Guide (Bicep Edition)

## Phase 1 — Prerequisites (one-time setup)

### Step 1: Install tools
Ensure you have the necessary tools installed on your local machine.

```bash
# 1. Azure CLI
winget install Microsoft.AzureCLI

# 2. kubectl
az aks install-cli

# 3. Docker Desktop (for building images)
# Download from docker.com

# 4. Verify everything works
az --version
kubectl version --client
docker --version
```

### Step 2: Login to Azure
Authenticate with your Azure account and select your subscription.

```bash
az login
```

### Step 3: Create a Resource Group
Create the resource group where your Bicep file will deploy all resources. We are using `centralindia`.

```bash
az group create --name online-boutique-rg --location centralindia
```

### Step 4: Deploy Infrastructure via Azure Bicep
Instead of running individual commands for AKS, ACR, Key Vault, and PostgreSQL, we will deploy them all at once using your Bicep file (`main.bicep`). 

*Note: You will be prompted to enter a secure password for the PostgreSQL database (`dbAdminPassword`).*

```bash
# Deploy all infrastructure
az deployment group create \
  --resource-group online-boutique-rg \
  --template-file main.bicep

# IMPORTANT: After the deployment finishes, note down the "Outputs" printed in your terminal. 
# You will need the ACR Name, Postgres FQDN, Key Vault Name, and Managed Identity Client ID for the next steps!
```

### Step 5: Connect to AKS & ACR
Now that the infrastructure is provisioned, log in to your Container Registry and connect your local `kubectl` to the new AKS cluster.

```bash
# Replace <acr-name-from-output> with your generated ACR name
az acr login --name <acr-name-from-output>

# Connect kubectl to your cluster
az aks get-credentials \
  --resource-group online-boutique-rg \
  --name online-boutique-aks

# Verify connection
kubectl get nodes
```

## Phase 2 — Build & Push Docker Images

Each service has its own Dockerfile. You need to build and push them all to your new Azure Container Registry.

### Step 6: Build and push all images

```bash
# Replace this with the ACR name generated by your Bicep deployment
ACR_NAME="<acr-name-from-output>"
ACR_URL="$ACR_NAME.azurecr.io"

# From project root — repeat for each service:
SERVICES=(cart-service catalog-service checkout-service currency-service email-service frontend-service payment-service recommendation-service shipping-service)

for svc in "${SERVICES[@]}"; do
  docker build -t "$ACR_URL/$svc:latest" "src/$svc"
  docker push "$ACR_URL/$svc:latest"
done
```
*Tip: You can deploy just 1-2 services first (see Phase 4 below).*

## Phase 3 — Configure AKS-specific settings

### Step 7: Fill in the placeholder values in AKS patches
Your Bicep file automatically provisioned your managed identities and databases. Open `deployments/k8s/overlays/aks/catalog-service-patch.yaml` and replace the placeholders with your **Bicep Outputs**:

* `<your-server>` → your PostgreSQL server FQDN (e.g., `pg-boutique-xxxxx.postgres.database.azure.com`)
* `<managed-identity-name>` → your Azure Managed Identity **Client ID** (from Bicep outputs)
* *(Optional)* `<your-cosmos-account>` → your CosmosDB account name (if using CosmosDB instead of Postgres for cart-service)

### Step 8: Workload Identity (Automated!)
**Skipped:** Your Bicep file *already* created the Managed Identity (`online-boutique-identity`) and the Federated Identity Credentials. You do not need to run any manual CLI commands for this!

### Step 9: Store your Stripe Secret in Key Vault
Your Bicep file *already* created the Key Vault and granted the Managed Identity access to it. You just need to add the actual secret.

```bash
# Replace <keyvault-name-from-output> with the KV name from your Bicep deployment
az keyvault secret set \
  --vault-name <keyvault-name-from-output> \
  --name stripe-secret-key \
  --value "sk_test_..."
```

### Step 10: Update image references in kustomization
Create/edit `deployments/k8s/overlays/aks/kustomization.yaml` to point to your new ACR.

```yaml
# Add this section to the existing kustomization.yaml
images:
  - name: frontend-service
    newName: <acr-name-from-output>.azurecr.io/frontend-service
    newTag: latest
  - name: cart-service
    newName: <acr-name-from-output>.azurecr.io/cart-service
    newTag: latest
  # ... repeat for all services
```

## Phase 4 — Deploy (incremental approach)

Yes, you can absolutely deploy parts at a time! Here's how:

### Option A: Deploy a single service manually
```bash
kubectl apply -f deployments/k8s/base/frontend-service/
```

### Option B: Deploy core services first (no external dependencies)
These services have no Azure-specific dependencies (Postgres/Key Vault) and are great for initial testing:
```bash
kubectl apply -f deployments/k8s/base/currency-service/
kubectl apply -f deployments/k8s/base/shipping-service/
kubectl apply -f deployments/k8s/base/recommendation-service/
kubectl apply -f deployments/k8s/base/cart-service/
kubectl apply -f deployments/k8s/base/envoy/
```

### Option C: Deploy everything via Kustomize (full AKS overlay)
```bash
kubectl apply -k deployments/k8s/overlays/aks/
```

```bash

kubectl get service frontend-service
```
*Wait for `EXTERNAL-IP` to appear, then open it in your browser.*

## Recommended Beginner Order
1. **Step 1-4** → Run Bicep to deploy full Infrastructure (one-time, ~15-20 min).
2. **Step 5-6** → Build images for simple services first (`currency`, `shipping`, `recommendation`).
3. **Option B** → Deploy those 3 services to AKS, verify they run (`kubectl get pods`).
4. **Step 7-9** → Update YAML patches with Bicep outputs (DB, Key Vault) and add your Stripe secret.
5. **Step 6** → Build remaining services.
6. **Option C** → Deploy everything.

## Common issues to watch for

| Problem | Fix |
| :--- | :--- |
| **ImagePullBackOff** | You forgot to update `kustomization.yaml` with your exact ACR URL, or you didn't run the `docker push` step. Bicep already handled the ACR-Pull permissions. |
| **CrashLoopBackOff** | Check logs: `kubectl logs <pod>`. Usually indicates a missing connection string, missing Key Vault secret, or wrong database password. |
| **Pending pod** | Not enough nodes. Since you are using B2s/D2s instances, you might need to scale up your Node Count in the Bicep file if you deploy all 11 microservices heavily. |
| **Frontend has no EXTERNAL-IP** | Wait 2-3 min; Azure LoadBalancer provisioning takes time. |
